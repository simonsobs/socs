UKKIDcontroller agent - installation and tutorial
=================================================

## Introduction

The UKKID controller agent, ukkid_controller.py is an OCS agent that
communicates with Knowledge Resources KRM RF System-on-chip (RFSoC)
boards which run the software and firmware for multi-tone readout of
the UK Kinetic Inductance Detectors (UKKIDs).

The RFSoC boards feature DACs, ADCs, FPGA fabric and a four-core ARM
processor running Ubuntu Linux. The tones to readout the KIDs are
generated by firmware pipelines running on the FPGA
fabric. Communication with these firmware pipelines is achieved by
python-based "readout server" software which runs on the RFSoCs ARM
cores. This server software takes IQ data which appears in the
firmware's/ARM cores shared memory and packages it into TCP/IP packets
which are sent to another computer which runs python software called
the "readout client". In practice this "readout client" software is a
python module, which can be imported into a python command line
interpreter and run interactively, or imported into other python
software or scripts. In our case the ukkid_controller.py imports the
readout_client module, which contains various methods for sending
instructions to and receiving data from the readout server software running
on the RFSoC's ARM cores.

 The ukkid_controller.py has processes and tasks with effectively act
 as wrapper functions (with appropriate logging and error handling)
 for the functions provided by the readout_client.py module, which is
 part of the souk_readout_tools package. These tasks/processes can,
 for example, request readout server configuration, start frequency
 scans, set tone frequencies and amplitudes and start streaming of raw
 IQ data. When streaming data for an astronomical observation, the
 ukkid_controller.py does not receive IQ TCP/IP data packets directly,
 but rather spawns separate streaming python streaming tasks (which
 also use the readout_client.py to communicate with the RFSoCs). The
 design is such that the ukkid_controller.py provides both
 sufficiently high-level tasks such that they can be used directly by
 telescope observers using the python "nextline" tool, and also
 low-level tasks for instrument characterisation and calibration,
 e.g. during commissioning. The ukkid_controller.py also provides
 tools and interfaces (to the souk_readout_tools resonator_fitter
 module) for fitting and extracting resonator parameters from
 frequency scans across the KID resonator arrays.


## Installation of prerequisites

 On a standard Ubuntu linux installation (at the time of writing
 Ubuntu 22.04 LTS) the following prerequisites are required.

1. An Observatory Control System (OCS) installation.

2. souk_readout_tools. This is the python code for the readout client
described above (note this repository also contains the code for the
readout server software that runs on the RFSoCs). **Use branch main**
for compatibility with the ukkid_controller agent.

3. The ukkid_controller agent itself - i.e. everything within the
agents/UKKIDcontroller directory. This can be obtained by checking out
the "ukkid_agent" branch of the official "socs" Simons Observatory
repository.

We will describe each of these in turn.

For this, one first needs to install docker (including docker

## Configuration of souk_readout_tools

Overall configuration can be maintained either on the server side
(i.e. on the RFSoC's Linux filespace) or on the client side (on your
computer's filespace). The latter is easier (and how the OCS
UKKID_controller agent does things), so follow the "create a config
from the bundled template" from the "quick-start" installation
instructions for souk_readout_tools. In brief, this involves invoking

```
from souk_readout_tools.config_utils import copy_template_config
copy_template_config('my_config_bun_p0.yaml', pipeline_id=0)
```

This will create a "my_config_bun_p0.yaml" file in the
souk_readout_tools directory.  You should edit it this to provide
e.g. the IP addresses you have set up for the RFSoCs on your
particular install, as described in the souk_readout_tools
documentation. The filename and location of this yaml file will need
to be provided to the ukkid_controller agent (see instructions for
this below).

is in ocs/ocs-site-configs), create a soft-link like...


configuration file set up for a particular OCS configuration scenario. E.g. mine contains

You should change the 'log-dir' element to point to a convenient

the hostname that gets returned by ```echo $HOSTNAME```, and the same
thing with ```-docker``` appended. This section

```
{'agent-class': 'UKKIDController',
```

declares that you will run up one instance of the UKKIDcontroller,
and that the command line argument ```--mode check_state``` and
```--kid_stream_id ufm_kid1``` will be passed to it behind the scenes.

## Tweaking UKKID_controller.py agent to work on your system.

(This is hacky at the moment and will be tidied up in the future.)

Change directory to where you have installed the ukkid_controller agent above. E.g.

```
cd /home/leechj/socs/socs/agents/ukkid_controller
```

Open in a text editor and change the following variables to the appropriate locations on your computer. You should only need to edit the variables
souk_readout_tools_install_dir and config_file_dict. The latter should be set up to match that set up in the section "Configuration of souk_readout_tools" above.

```
##################
# Set to where you installed souk_readout_tools
souk_readout_tools_install_dir = '/home/leechj/souk_readout_tools'
#############
# Set to True to run without RFSoC hardware i.e. in simulated mode for testing.
mock = True
#############
# Where the site packages have been installed by the python virtual enivroment - should contain a dir called souk_readout_tools
souk_readout_tools_path = os.path.join(souk_readout_tools_install_dir,'client_venv/lib/python3.10/site-packages')
# Should contain the script client_scripts/receive_stream_g3.py
stream_script_path =  os.path.join(souk_readout_tools_install_dir,'src/souk_readout_tools/client/')
# Output directory where you want your data files to get written to
data_out_dir = os.path.join(souk_readout_tools_install_dir,'src/souk_readout_tools/client/client_scripts/tmp/')
# Directory where the config files specified in the config_file_dict dictionary below are located.
config_file_root_dir = souk_readout_tools_install_dir
# This next variable tells the agent which config file to push to the server depending on which kid_stream_id the agent was started with on the command line.
# This will eventually have 28 entries for the 28 individual reduction pipelines.
# For single pipline running, with one ukkid_controller.py agent running at a time, we only need the first entry, and it should the filename of
# your souk_readout_tools config .yaml file which lives in config_file_root_dir specified above.
config_file_dict = {'ufm_kid1':'my_config_bun_p0.yaml','ufm_kid2':'my_config_bun_p1.yaml'}
##################
```

If you set the mock variable to be True, the agent will run in simulated mode and you do not need a connected hardware RFSoC. Set to False if you want to run
real hardware. In that case the IP address etc. should be set up in souk_readout_tools config.yaml file specfied in config_file_dict.

### 7) Go back to the python terminal, try a standard "cold-start" observing procedure.

From the python terminal in (4) above, do...


The output data files (.csv, .png, .g3) will appear in where you set the variable data_out_dir.

data_out_dir = os.path.join(souk_readout_tools_install_dir,'src/souk_readout_tools/client/client_scripts/tmp/')

```
ln -s /home/leechj/souk_readout_tools/src/souk_readout_tools/client/client_scripts/tmp/ ~/data_dir
```

Data is output into directories named after the first 5 digits of the
start Unix C-time in seconds, followed by the kid_stream_id which is passed
to the command line of the ukkid_controller, via the yaml file above e.g.

```
<data_dir>/17823/ufm_kid1/
```

There is a one-to-one relationship between the kid_stream_id string,
an instance of a ukkid_controller agent, a server/firmware pipeline on
the RFSoC and a particular channel on a particular KID wafer on the
focal plane.

More details on what each of these "cold-start" tasks is doing, and
what output to expect, can be found in the Software ICD here.

https://www.overleaf.com/read/djpysbgvydsw#875792

## Running the UKKIDcontroller agent with real RFSoC hardware

The procedure to run the UKKIDcontroller agent with a real RFSoC
connected to your network is identical to that when running in mocked
mode - just set "mock = False" in the ukkid_controller.py file. You
should ensure the yaml files specifiecid in

```
config_file_dict = {'ufm_kid1':'my_config_bun_p0.yaml','ufm_kid2':'my_config_bun_p1.yaml'}
```

Are correctly set up with the ip address of the RFSoc etc. You should
hopefully have already gone through this process when installing and
verifying your souk_readout_tools installation before you set up the
OCS system and the ukkid_controller agent.
